## 2. 代码结构与命名规范分析

### 2.1 目录结构约定

- **组件目录 (`components/`)**: 按功能模块组织，如 `interact/`、`rtc/`、`common/` 等
- **页面目录 (`pages/`)**: 存放页面级组件
- **状态管理 (`stores/`)**: 采用模块化结构，每个功能模块对应一个store
- **服务层 (`server/`)**: 按业务模块划分API请求
- **工具函数 (`utils/`)**: 通用功能的工具函数
- **类型定义 (`types/`)**: 集中管理TypeScript类型

### 2.2 文件命名风格

- **组件文件**: 采用 `kebab-case`，如 `header-bar.vue`、`menu.vue`
- **工具文件**: 采用 `camelCase`，如 `localstorage.js`、`broadcast.js`
- **TypeScript文件**: 使用 `.ts` 扩展名，命名采用 `camelCase` 或 `kebab-case`
- **样式文件**: 组件内样式文件多命名为 `index.less`，与组件同名

### 2.3 组件命名风格

- **组件名**: 在模板中使用 `kebab-case`，如 `<header-bar>`、`<push-video>`
- **导入名**: 在脚本中使用 `PascalCase`，如 `import HeaderBar from '@/components/record/headerBar.vue'`
- **组合式API**: 使用 `script setup` 语法

### 2.4 变量/函数命名风格

- **变量名**: 采用 `camelCase`，如 `uploadVideoStatus`、`percentage`
- **常量名**: 使用 `PascalCase` 或全大写，如 `ClassModel`、`InteractType`
- **函数名**: 采用 `camelCase`，如 `createChildrenWindow`、`onStage`
- **Pinia store**: 采用 `useXxxStore` 命名模式，如 `useInteractStore`、`useVideoUploadStore`

### 4.3 状态管理模式

**Pinia Store 模式 (`interact.ts`)：**

```typescript
import { computed, ref } from 'vue'
import { defineStore } from 'pinia'
import type { DataType } from '@/types'

export const useSomeStore = defineStore('storeName', () => {
  // 1. 定义响应式状态
  const loadingData = ref({
    show: false,
    text: '加载中，请稍后...'
  })
  
  const currentData = ref<DataType | null>()
  
  // 2. 定义计算属性
  const derivedValue = computed(() => {
    return currentData.value ? someTransformation(currentData.value) : null
  })
  
  // 3. 定义操作方法
  function updateData(newData: DataType) {
    currentData.value = newData
  }
  
  // 4. 返回状态和方法
  return {
    loadingData,
    currentData,
    derivedValue,
    updateData
  }
})
```

## 四、通用组件使用最佳实践

### 1. 消息提示使用规范
- 重要操作使用 `confirm` 进行二次确认
- 操作结果反馈使用 `toast`
- 临时提示信息使用 `popTip`

### 2. 表单组件使用规范
- 统一使用组件库中的表单组件，保持样式一致性
- 根据功能需要选择合适的组件类型
- 表单状态管理推荐使用 Vue 的响应式系统

### 3. 性能优化建议
- 动态创建的组件（如消息提示）在使用完毕后及时销毁
- 避免在组件中添加过多不必要的事件监听器
- 对于频繁使用的组件，考虑使用缓存策略

### 4. 样式一致性维护
- 通用组件的样式统一管理在 `/src/components/common/form/index.less` 等文件中
- 组件样式应遵循项目的设计规范
- 使用 CSS 变量或 LESS 变量管理主题颜色

## 五、注意事项

1. 组件导入路径使用 `@` 别名指向 `src` 目录
2. 部分组件使用了 Options API，部分使用了 Composition API，注意兼容使用
3. 消息相关组件会自动处理组件的创建和销毁，无需手动管理生命周期
