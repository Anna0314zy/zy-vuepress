这个项目没有用 Redux / Zustand 这类传统状态管理库，数据分三个层次来管：
# 数据
---

## 第一层：引擎数据（核心）

课件内容存在 **MobX 响应式的 TreeNode 树**里，由 `@editor/core` 的 Engine 管理：

```
Engine
 └── Workbench
      └── Workspace[]（每页课件一个）
           └── TreeNode（Root）
                ├── props.style       // 样式
                ├── props.animates[]  // 动画配置
                └── children[]
                     ├── TreeNode（图片组件）
                     ├── TreeNode（文本组件）
                     └── ...
```

**改任何属性直接 `node.props.xxx = value`，MobX 自动触发视图更新，不需要手动 dispatch。**

---

## 第二层：React 状态（页面级）

`App.tsx` 里用 `useState` 管理编辑器级别的 UI 状态：

| 状态 | 作用 |
|------|------|
| `workspaceList` | 左侧缩略图列表（页面顺序） |
| `currentWorkspaceId` | 当前激活的页面 |
| `globalResource` | 全局资源文件列表 |
| `globalData` | 视口比例、COS 配置等全局配置 |
| `saveText` | 顶部保存状态提示文字 |

---

## 第三层：全局共享状态

两种方式跨组件共享数据：

| 方式 | 用在哪 |
|------|--------|
| `React.createContext` | `GlobalDataContext`（视口/COS配置）、`GlobalResourceContext`（资源列表）向深层组件传递 |
| `hox` | 轻量全局状态，管理实例 Map（`instanceMap`，存每个组件的运行时实例引用） |

---

## 数据流向总结

```
用户操作
  ↓
直接修改 TreeNode.props（引擎数据）
  ↓
MobX 响应式 → 画布组件自动重渲染

切换页面 / 顺序变化
  ↓
setWorkspaceList / setCurrentWorkspaceId（React state）
  ↓
触发 useEffect → 同步引擎 Workbench 状态

保存
  ↓
workspace.serialize() 序列化 TreeNode 树为 JSON
  ↓
POST 到服务端
```

---

**一句话总结**：课件内容数据用 MobX 管（改了自动更新），页面导航/列表用 React useState 管，跨层级共享用 Context 和 hox。三者各司其职，没有统一的全局 store。
每一层解决不同的问题，所以分开管：

---

## MobX 管课件内容

课件内容（组件的位置、样式、动画）**改动非常频繁**——拖拽时每一帧都在改坐标，如果用 React setState 每次改都会触发整个组件树重渲染，性能会很差。

MobX 的优势是**精准更新**：只有真正用到这个数据的组件才会重渲染，其他组件不受影响。所以拖拽时只有被拖动的那个组件在更新，其他组件完全不动。

---

## React useState 管页面列表

页面列表（顺序、当前激活哪页）是**编辑器级别的 UI 状态**，和课件内容本身无关，改动频率低，用 React 自己的 useState 管理最自然，也不需要引入额外的复杂度。

---

## Context 管全局配置

视口比例、COS 上传配置这类数据**很少变化，但很多层级深的组件都需要用**，用 Context 往下透传比一层一层 props 传方便得多。

---

## hox 管实例 Map

游戏、视频等组件运行时需要保存一个实例引用（比如控制游戏暂停/播放），这个引用**既不是 UI 状态也不是课件数据**，放 MobX 或 useState 都不合适，hox 作为轻量全局容器刚好够用。

---

**一句话**：数据改动频率越高、越需要精准更新的用 MobX；跨层级共享但变化少的用 Context；编辑器 UI 本身的状态用 useState。没有用一个大 store 统一管理，是因为不同数据的特性差异太大，强行统一反而复杂。
