# Editor 项目与服务端交互数据分析

### 总体策略：**页面级全量替换**（本地 Diff 优化）

不是整个课件全量更新，而是**以单个页面（slide）为单位做全量替换**。

---

### 数据流程

```
用户编辑 widget / asset / animation
        ↓
Redux action 触发（store/models/project.ts）
        ↓
realTimeSave 中间件拦截（store/middlewares/realTimeSave.ts）
        ↓  debounce 1000ms
savePageJson → updatePageJson（services/projectService/pageStore.ts）
        ↓
1. clearPageJson：清理纯 UI 字段（hidden、text、shouldCreateImage 等）
2. 与 IndexedDB 本地缓存做 JSON.stringify 对比 diff
3. 有变化才 → gzip 压缩 → POST /material/pageJson/compress/create
```

---

### 核心数据结构

**保存单页的请求体：**
```typescript
{
  icwCode: string,        // 课件唯一标识
  materialCode: string,   // 页面唯一标识
  data: string,           // gzip 压缩后的完整页面 JSON（compressVersion: 1）
  creatorId: string,
  tag: string,            // 如 "animation,answer,voice"
  compressVersion: 1
}
```

**压缩前的完整页面 JSON（Page 类型）：**
```typescript
{
  version: string,
  widgets: WidgetInstance[],   // 该页所有 widget 数据
  assets: Asset[],             // 仅保留实际使用的素材
  answers: AnswerItem[],
  animation: Animations[],
  animationGroups: [...],
  interactions: Interaction[],
  logic: { blocks, links, vars },
  options: PageOptions,
  animationMicro: [...]
}
```

**初始化加载：一次请求返回全部数据**
```typescript
// GET /icw/init/v1
{
  editorIcwVo: { title, icwCode, latestUuid, ... },
  materialVoList: [{ code, json, orderNum, title, ... }],  // 所有页面
  editorSectionVoList: Section[],
  manifestVo: { status, uuid, ... },
  editorAssetVos: GameAsset[]
}
```

---

### 各维度的全量 vs 部分更新对比

| 操作维度 | 策略 | 接口 |
|---|---|---|
| **页面内容保存** | **页面级全量替换** | `/material/pageJson/compress/create` |
| **多页批量保存** | 全量批次 | `/material/pageJson/compress/batch/create` |
| **页面排序** | 全量列表替换 | `/material/order`（发所有页的 orderNum） |
| **章节（Section）** | 全量列表替换 | `/section/submit`（发所有 section） |
| **页面元数据** | 部分更新 | `/material/page/update`（只发 title/duration/notes） |
| **初始加载** | 一次全量拉取 | `/icw/init/v1` |

---

### 这样设计的好处

1. **服务端逻辑极简** — 服务端只需存储客户端发来的内容，无需做复杂 merge，冲突处理是"最后写入胜"。

2. **本地 Diff 减少无效请求** — `updatePageJson` 中通过 `JSON.stringify` 与 IndexedDB 缓存对比，只有真正发生变化才发请求，加上 1000ms debounce，大幅降低请求量。

3. **Gzip 压缩降低传输体积** — `gzipStr()` 在发送前压缩 JSON，代码注释中曾打印压缩率，实际效果显著。

4. **保存前做 Schema 校验** — `validate(state)` 在每次上传前验证数据合法性，避免脏数据写入服务端。

5. **IndexedDB 作为本地数据源** — `localforage`（`Editor_Pages_Cache`）既是缓存也是 diff 基准，页面刷新后可恢复状态，并减少服务端调用。

6. **BroadcastChannel 跨 Tab 同步** — 保存时通过 `new BroadcastChannel('pageJsonUpdate')` 广播给预览 iframe 等其他窗口，无需额外服务端请求。

---

### 未来可以改造的方向

1. **页面 JSON 过大时负载高** — 当前每次保存发整个页面 JSON（widget 多时很大）。可引入 **JSON Patch（RFC 6902）** 做 widget 级别的差量更新，大幅降低 payload 体积。

2. **章节/排序用全量替换** — `submitSections` 每次排序都发所有章节，可改为 `{ sectionCode, newOrderNum }` 的单条 patch 接口。

3. **没有乐观锁 / 并发控制** — 保存请求中没有版本号或 `lastModifiedTime` 校验，多人同时编辑会静默覆盖。引入 `optimisticLock` 字段可为未来协同编辑打基础。

4. **初始化全量加载所有页** — `materialVoList` 把所有页的 JSON 一次性返回，课件页数多时初始 payload 很大。可改造为**按需懒加载**（打开哪页才请求那页的 JSON）。

5. **JSON.stringify diff 不稳定** — 对象键顺序在不同操作后可能改变，导致误判为"有变化"而发出无意义请求。建议换用 `fast-deep-equal` 等结构化深比较库。

6. **无实时协同** — 当前是纯轮询/防抖模式。项目中已内置了 socket.io（`Main/socket.io.js`），可以此为基础扩展 WebSocket 实时同步，支持多人协作编辑。

7. **Tag 计算在客户端硬编码** — `animation/answer/voice` 等 tag 由客户端代码计算，后续新增类型需改前端。可改为服务端根据 JSON 内容自动推断 tag。
