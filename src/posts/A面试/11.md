## Serverless 截图服务 — 详细说明

### 一、它是干什么的？

这是一个**幻灯片页面截图服务**，部署在阿里云函数计算（FC）上。其核心职能是：

> 接收幻灯片页面数据（JSON 格式），通过 Puppeteer 启动无头浏览器访问预渲染页面，等待页面渲染完成后截取 `#preview` 区域的截图，上传到腾讯云 COS（对象存储），最后回调后端 API 将截图路径写回业务系统。

**整个数据流：**

```
业务后端 → HTTP POST → 阿里云FC函数 → Puppeteer → slide-shot渲染页 → 截图 → 上传COS → 回调业务后端
```

### 二、技术亮点

#### 1. 浏览器实例复用（单例模式 + 初始化钩子）

[setup.js](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/core/setup.js) 使用 IIFE 闭包实现了浏览器单例：

- 在 FC 的 [`initializer`](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/index.js#L20-L24) 钩子中预启动浏览器，避免每次请求冷启动都重新 launch
- 通过 `isBrowserAvailable()` 检测浏览器是否存活，崩溃时自动重建
- 使用 `Starting` 标志位防止并发启动，等待中的请求通过轮询等待

这是函数计算场景下的经典优化——FC 实例可复用时，浏览器已在内存中，后续调用无需再 launch。

#### 2. 变量注入（`injectVariable`）而非 URL 传参

[index.js#L26-L37](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/index.js#L26-L37) 使用 `page.evaluateOnNewDocument` + `Object.defineProperty` 在页面加载前将数据注入到 `window.json`：

```js
Object.defineProperty(window, key, {
  get: function () { return value; }
});
```

这比 URL 拼接参数更安全，避免了大数据量时 URL 长度限制和编码问题。

#### 3. 前后端协作的截图时机控制

截图不是简单的 `page.goto()` + `page.screenshot()`，而是一个**前后端协作的双向通信机制**：

- **后端**通过 `page.exposeFunction("generateScreenShot", callback)` 向页面暴露一个全局函数
- **前端** [slide-shot/App.tsx](file:///Users/zouyu/Desktop/ledu/slides-engine/common/slide-shot/src/App.tsx) 在资源全部加载完成后调用 `window.generateScreenShot()`
- **后端**在 callback 中等待 `#preview` 元素出现，再精确截取该 DOM 元素

这解决了"何时截图"的核心问题——必须等图片等异步资源加载完毕才能截图，否则截图是空白的。

#### 4. 资源加载检测（前端侧）

[page.ts](file:///Users/zouyu/Desktop/ledu/slides-engine/common/slide-shot/src/page.ts) 实现了资源加载检测：

- 递归遍历组件树，收集所有 `Img` / `Video` 组件
- 轮询检测 DOM 中 `<img>` 元素的 `img.complete` 属性
- 额外检测背景图（通过 `getComputedStyle` 取 `background-image`，手动 `new Image()` + `onload`）
- 全部加载完成后才触发截图

#### 5. 页面加载重试机制

[index.js#L102-L114](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/index.js#L102-L114) 对 `page.goto()` 做了最多 6 次重试，应对函数计算环境中网络不稳定的场景。

#### 6. Puppeteer 在 FC 环境的特殊配置

[config.js](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/core/config.js) 专门适配了阿里云 FC 的受限运行环境：

- `--no-sandbox`：FC 没有 sandbox 权限
- `--disable-gpu`：FC 没有 GPU，避免加载 libosmesa 报错
- `--no-zygote`：避免多进程问题导致 freeze
- 使用阿里云官方 Layer `Nodejs-Puppeteer17x` 提供 Chromium 运行时

#### 7. 多环境部署配置

三套 YAML 配置对应不同环境：

| 文件 | 环境 | 关键差异 |
|------|------|----------|
| `s.yaml` | 默认/开发 | 无 `SHOT_URL` 环境变量 |
| `s.fz.yaml` | 测试 | `SHOT_URL` 指向测试环境 slide-shot |
| `s.prod.yaml` | 生产 | 函数名 `slide_screenshot_prod`，`SHOT_URL` 指向生产（实际 prod 配置中 URL 应替换为生产域名） |

### 三、实现流程完整梳理

```
1. FC 实例初始化
   └── initializer 钩子调用 setup.getBrowser() 启动 Puppeteer 浏览器

2. 业务系统发送 HTTP POST 请求
   ├── headers 中携带 pageSnapshotId（页面快照ID）
   └── body 中携带页面 JSON 数据（包含组件树结构、文件资源等）

3. handler 函数处理请求
   ├── 通过 raw-body 读取请求体（限制 10MB）
   ├── 解析 JSON body
   └── 调用 screenshot() 函数

4. screenshot 函数执行
   ├── 获取浏览器单例
   ├── 创建新页面，设置视口 1300×1300
   ├── 注入 window.json（页面数据）
   ├── 注册 window.generateScreenShot 回调
   │   └── 等待 #preview 选择器 → 截取元素 → 上传 COS → 回调业务API
   └── page.goto() 访问 slide-shot 页面（带重试）

5. slide-shot 前端页面渲染
   ├── 读取 window.json 获取页面数据
   ├── 使用 RenderRoot 组件渲染幻灯片
   ├── Page 类追踪资源加载状态（图片 complete 检测）
   ├── 等待所有图片/背景图加载完毕
   └── 调用 window.generateScreenShot() 通知后端

6. 后端回调执行
   ├── 等待 #preview DOM 元素出现
   ├── 截取 #preview 元素为 PNG buffer
   ├── 关闭页面
   ├── 上传到 COS（路径: /slides/resources/screenshot/{uuid}.png）
   └── POST 回调业务后端 /classroom-slides/node/callback/page-screenshot
       携带 pageSnapshotId + screenshotOssPath

7. 返回 HTTP 响应 "success"
```

### 总结

这是一个典型的 **Serverless + Headless Browser + 对象存储** 的服务端截图方案。亮点在于：浏览器实例复用优化冷启动、前后端双向通信精确控制截图时机、资源加载完整性检测保证截图质量、以及针对 FC 受限环境的适配配置。

基于上一轮对 serverless 截图服务的完整代码分析，下面从**技术难点**和**面试官可能追问**两个维度梳理：

---

## 一、技术难点

### 难点 1：函数计算环境下运行浏览器的适配

阿里云 FC 是受限沙箱环境，默认无法运行 Chromium。需要解决：

- 没有 sandbox 权限 → `--no-sandbox` + `--disable-setuid-sandbox`
- 没有 GPU → `--disable-gpu`，否则加载 `libosmesa.so` 报错
- 多进程模型在容器中可能 freeze → `--no-zygote`（注意 `--single-process` 被注释掉了，因为它会导致 `newPage()` 卡死）
- Chromium 二进制体积大 → 依赖阿里云官方 Layer `Nodejs-Puppeteer17x`，运行时通过 `LD_LIBRARY_PATH` 和 `NODE_PATH` 指向 Layer 路径

**面试可能追问：**
> 为什么 `--single-process` 会卡死？`--no-zygote` 和 `--single-process` 有什么区别？

`--single-process` 让浏览器和渲染器在同一进程运行，在 FC 的 container 中容易死锁（信号处理冲突）。`--no-zygote` 只是跳过 zygote 预 fork 进程，渲染器仍独立进程，更稳定。

---

### 难点 2：截图时机的精确控制

最核心的难点：**什么时候截？截早了资源没加载完是白图，截晚了浪费 FC 执行时间（按毫秒计费）。**

解决方案是前后端双向通信：

- 后端通过 `page.exposeFunction` 向浏览器上下文注入 `generateScreenShot`
- 前端在资源加载完成后主动调用 `window.generateScreenShot()`
- 但前端的"资源加载完成"判断本身很复杂：
  - 普通图片：`img.complete` 轮询检测
  - CSS 背景图：`getComputedStyle` → 手动 `new Image()` + `onload`
  - 视频：用视频封面图替代（不渲染视频本身）

**面试可能追问：**
> 为什么不用 `page.waitForNetworkIdle()` 或者 `window.onload`？

- `waitForNetworkIdle` 认为网络静止 500ms 就算完成，但图片可能还在解码渲染到屏幕，截出来可能还是空白
- `window.onload` 不等待动态创建的 `<img>` 和 CSS 背景图
- 自定义的资源检测机制最可靠，因为知道业务中哪些组件需要等待

---

### 难点 3：浏览器实例生命周期管理

函数计算实例会被回收和复用，浏览器进程可能崩溃：

- **实例复用**：用 IIFE 闭包 + `initializer` 钩子预启动浏览器，避免每次请求冷启动（Puppeteer launch 需要 2-5 秒）
- **崩溃检测**：每次 `getBrowser()` 调用 `isBrowserAvailable()` 通过 `browser.version()` 探活
- **并发启动防抖**：`Starting` 标志位 + 轮询等待，防止多个并发请求同时启动浏览器

**面试可能追问：**
> 浏览器崩溃后重建期间，并发请求怎么处理？

当前实现用 `setInterval` 轮询等待 `browser` 变量有值。这个方案有隐患：
- 如果浏览器启动失败，没有 `reject` 逻辑，会永远轮询
- 更好的方案是用 Promise + 事件通知机制替代轮询

---

### 难点 4：FC 冷启动与性能优化

FC 按执行时间计费，冷启动链路长：

```
FC冷启动 → 加载Layer → initializer启动浏览器 → 请求到达 → 打开页面 → 等待渲染 → 截图 → 上传
```

优化措施：
- `initializer` 钩子预启动浏览器，

当前代码的问题在 [setup.js#L25-L36](file:///Users/zouyu/Desktop/ledu/slides-engine/serverless/screenshot/code/core/setup.js#L25-L36)：

```js
// 问题：无超时、setInterval 未清除、启动失败时永远等不到
if (Starting) {
  await new Promise((resolve, reject) => {
    setInterval(() => {
      if (browser) { resolve(); }  // 永远不会 reject
    }, 10);
  });
}
```

三个缺陷：
1. 浏览器启动失败 → 没有 reject，永远轮询
2. `setInterval` resolve 后没 `clearInterval`，内存泄漏
3. 没有超时，请求可能挂死

## 改进方案：Promise + 事件通知
改动已完成。下面是核心思路解释：

---

## 方案对比

### 旧方案（轮询）

```
请求A: Starting=true → puppeteer.launch()
请求B: Starting=true? → setInterval 每10ms检查 browser 是否有值
请求C: 同上，也在轮询
```

问题：轮询浪费 CPU、无超时、无错误传播、`setInterval` 泄漏。

### 新方案（Promise 共享）

```
请求A: launchPromise = null? → 创建 launchPromise = puppeteer.launch() → await launchPromise
请求B: launchPromise 已有? → 直接 await launchPromise（同一个 Promise）
请求C: 同上
```

**核心思想：** `launchPromise` 本身就是一个"事件"。所有等待者 `await` 同一个 Promise，启动完成时所有等待者同时被唤醒，不需要轮询。

---

## 关键设计点

| 场景 | 处理方式 |
|------|----------|
| 首次调用 | `launchPromise` 为 null，创建并 await |
| 并发调用 | `launchPromise` 已有，直接 await 同一个 Promise |
| 启动成功 | `.then` 赋值 `browser`，所有等待者拿到同一个实例 |
| 启动失败 | `.catch` 清空 `launchPromise`，抛出错误给所有等待者；后续请求可重新尝试启动 |
| 浏览器崩溃 | 下次 `isBrowserAvailable` 返回 false，走重新启动逻辑 |
| 启动完成后 | `.finally` 清空 `launchPromise`，确保崩溃重建不受旧 Promise 影响 |

这是典型的 **Promise 缓存模式（Promise Cache / Singleton Promise）**，在并发场景下保证只有一个启动过程，同时所有调用方都能正确拿到结果或错误。
