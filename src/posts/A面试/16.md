

# 信令驱动播放的实现机制

这个幻灯片引擎使用了一个完整的**信令系统**来实现播放控制，主要包含以下几个核心组件:

### 1. **信令架构核心**

#### [Signalling 类](file:///Users/zouyu/zouyu/生产系统/slides-engine/common/render-core/components/EventSequence.tsx#L35-L165)
这是信令系统的核心控制器，负责:
- **发送端 (Sender)**: 定时检查消息队列并发送信令
- **接收端 (Receiver)**: 监听并处理接收到的信令
- **角色管理**: 支持动态切换 Sender/Receiver 角色

```typescript
// 发送端逻辑 - 定时检查并发送消息队列中的信令
async check() {
  const { msgQueue } = getEventStore();
  const sendQueue = msgQueue.filter(item => !item.marked && item.msgType !== "state");
  if (sendQueue.length > 0) {
    for (const item of sendQueue) {
      await this.messageClient.sendWithRetry(item);
    }
  }
}

// 接收端逻辑 - 接收信令并执行对应控制器
receive() {
  this.messageClient.onMessage((msgInfo) => {
    const filterList = msgControllerList.filter(/* 匹配条件 */);
    const item = filterList[filterList.length - 1];
    const result = item?.controller?.(msgInfo.msgDetail);
    Promise.resolve(result).then(() => {
      addMsg(msgInfo);
    });
  });
}
```

### 2. **消息传输层**

#### [PostMessageClient](file:///Users/zouyu/zouyu/生产系统/slides-engine/common/render-core/components/PostMessageClient.ts#L4-L85)
负责实际的信令传输，支持两种模式:

- **预览模式 (preview)**: 使用 `BroadcastChannel` 进行本地广播
- **生产模式**: 通过 `window.microApp.forceDispatch` 与原生桥接通信

```typescript
// 发送消息
public sendWithRetry(message: any) {
  if(this.isPreview){
    this.channel.postMessage(message);
  } else {
    const data = {type:CoursewareEvent.TransferMessageSend, param:message, timestamp: message.timeStamp};
    this.channel.sendMessage(data);
    message.marked = true;
  }
}

// 接收消息
public onMessage(callback: (message: any) => void) {
  this.listeners.push(callback);
}
```

### 3. **消息队列管理**

#### [EventStore (Hox)](file:///Users/zouyu/zouyu/生产系统/slides-engine/common/render-core/models/context.ts#L158-L238)
使用 Hox 全局状态管理，维护两个核心数据结构:

- **msgQueue**: 消息序列列表，记录所有发送和接收的消息
- **msgControllerList**: 组件注册的消息处理器列表

```typescript
const registerMsg = useCallback((id, msgName, msgType, pageId) => ({
  // 发送端调用: 触发信令发送
  notice(msgDetail) {
    addMsg({ id, msgName, msgType, msgDetail, pageId }, true);
  },
  
  // 接收端调用: 注册消息处理逻辑
  register(controller) {
    // 先检查是否有缓存的断线重连消息
    const cache = msgStore.getCache();
    const cacheInfo = cache.find(/* 匹配条件 */);
    if(cacheInfo) {
      controller(cacheInfo.msgDetail);
    }
    setMsgControllerList((old) => [
      ...old,
      { id, msgType, msgName, pageId, controller }
    ]);
  }
}), []);
```

### 4. **视频/音频组件的信令注册**

以[视频组件](file:///Users/zouyu/zouyu/生产系统/slides-engine/common/render-components/src/video/index.tsx#L33-L677)为例:

#### 注册发送者 (Sender)
监听播放器事件，触发信令发送:

```typescript
function registerSender(playerEventNotice, playerStateNotice) {
  // 监听时间更新事件 (节流 5s)
  const syncTime = throttle(() => {
    const state = getPlayerState(PlayerEventTypeCollection.TIMEUPDATE);
    [playerStateNotice, playerEventNotice].forEach((notice) => {
      notice({
        type: noticeIndex === 0 ? PLAYERSTATE : PLAYEREVENT,
        payload: { state, event: PlayerEventTypeCollection.TIMEUPDATE }
      });
    });
  }, 5e3);
  
  videoEl.addEventListener(PlayerEventTypeCollection.TIMEUPDATE, syncTime);
  
  // 监听其他事件: play, pause, playing, error 等
  playerEvents.forEach((event) => {
    const handler = () => {
      const state = getPlayerState(event);
      [playerStateNotice, playerEventNotice].forEach((notice) => {
        notice({ type, payload: { state, event } });
      });
    };
    videoEl.addEventListener(event, handler);
  });
}
```

#### 注册接收者 (Receiver)
接收信令并恢复播放状态:

```typescript
function registerReceiver(playerEventRegister, playerStateRegister) {
  [playerEventRegister, playerStateRegister].forEach((register) => {
    register((msg) => {
      const { event, state } = msg.payload;
      const { reconnect, type } = msg;
      recoverPlayerState(state, reconnect, type); // 恢复播放状态
    });
  });
}

// 状态恢复逻辑
function recoverPlayerState(state, reconnect, changeType) {
  const { remotePaused, currentTime, playbackRate } = state;
  
  // 处理播放/暂停状态
  if (remotePaused !== null) {
    if (remotePaused) {
      playerRef.current.pause();
    } else {
      playerRef.current.play();
    }
  }
  
  // 处理播放进度
  if (currentTime !== null) {
    if (changeType === PlayerEventTypeCollection.TIMEUPDATE) {
      // timeupdate 类型只在断线重连/首次播放/时间差>3s 时设置
      if (reconnect || isFirstPlay || diff >= 3) {
        setCurrentTime(time, diff);
      }
    } else {
      // 非 timeupdate 类型直接设置
      setCurrentTime(time, diff, true);
    }
  }
  
  // 处理倍速
  if (playbackRate) {
    playerRef.current.playbackRate = playbackRate;
  }
}
```

### 5. **BindManager (绑定管理器)**

[BindManager](file:///Users/zouyu/zouyu/生产系统/slides-engine/common/bind-manager/src/model/bindManager.ts#L18-L325) 用于管理音频等媒体资源的信令绑定:

```typescript
// 初始化绑定
public addBind(bind: IBindData) {
  // 1. 注册信令 API
  const registerEventNotice = this.registerMsg(bind.id, bind.type, 'event', this.pageId);
  const registerStateNotice = this.registerMsg(bind.id, bind.type, 'state', this.pageId);
  
  // 2. 创建实例 (AudioInstance 等)
  this.registerInstance({
    registerEventNotice,
    registerStateNotice,
    // ... 其他参数
  });
  
  // 3. 注册触发器 (如 click 事件)
  setTimeout(() => {
    this.registerTrigger(bind);
  }, 100);
}

// 执行绑定动作
private executeBind(bind: IBindData) {
  const instance = this.bindInstanceMap.get(bind.id);
  switch (bind.type) {
    case EnumBindType.Audio:
      // 先暂停其他音频，避免信令顺序错乱
      audioInstances?.forEach((item) => {
        if(item?.id !== bind.id && !item?.audioElement?.paused) {
          item?.pause?.(true);
        }
      });
      instance.playPause?.(true);
      break;
  }
}
```

### 6. **完整工作流程**

```
发送端 (Sender)                        接收端 (Receiver)
    |                                       |
    | 1. 用户操作 (播放/暂停/seek)           |
    ↓                                       |
2. 播放器触发事件                           |
   (play/pause/timeupdate)                  |
    |                                       |
3. registerSender 监听事件                  |
    |                                       |
4. notice() 调用 → 加入 msgQueue           |
    |                                       |
5. Signalling.check() 定时扫描             |
    |                                       |
6. PostMessageClient.sendWithRetry()       |
    |--------------------------------------→|
    |         7. onMessage 接收信令          |
    |                                       ↓
    |                          8. 匹配 msgControllerList
    |                                       |
    |                          9. 执行 controller(msgDetail)
    |                                       |
    |                          10. recoverPlayerState()
    |                              - 设置播放/暂停
    |                              - 设置 currentTime
    |                              - 设置 playbackRate
    |                                       |
    |                          11. addMsg() 记录到队列
```

### 7. **关键特性**

1. **消息类型区分**:
   - `event`: 事件型消息 (play, pause 等瞬时动作)
   - `state`: 状态型消息 (currentTime, playbackRate 等持续状态)

2. **断线重连支持**:
   - 使用 `msgStore._cache` 缓存消息
   - 接收端注册时检查缓存并恢复状态

3. **防抖节流**:
   - timeupdate 事件 5 秒节流，避免频繁发送

4. **角色动态切换**:
   - 支持 Sender ↔ Receiver 切换
   - 切换时清理消息队列

5. **重试机制**:
   - `sendWithRetry` 支持发送重试

这套信令系统实现了**发送端操作实时同步到接收端**的功能，适用于在线教学场景中教师端控制学生端课件播放的需求。

# 为什么需要用到微前端?

根据代码分析,这个幻灯片引擎项目使用微前端架构(`@ld/micro-app`)主要有以下几个核心原因:

### 1. **多类型课件内容的动态加载与隔离**

项目需要支持多种不同类型的页面内容:

- **普通课件页** (pageType=1): 基于 React 的幻灯片内容
- **游戏页** (pageType=2): 基于 Cocos 引擎的游戏内容  
- **视频页** (pageType=3): 视频播放页面

```typescript
// mcc-player/src/App.ts - 动态加载不同的子应用
await microApp.renderApp({
  name: "course",  // 课件子应用
  url,             // 远程URL
  inline: true,
  container: this.mccContainer,
  lifeCycles: { /* ... */ }
});

await microApp.renderApp({
  name: "cocos",   // 游戏子应用  
  url,
  inline: true,
  container: this.mccContainer,
  disableSandbox: true,  // 游戏应用禁用沙箱
  destroy: true,
});
```

**为什么需要微前端:**
- Cocos 游戏是独立的游戏引擎,与 React 技术栈完全不同
- 不同课件可能有不同的技术实现,需要独立部署和更新
- 避免将所有代码打包到一起,减少主应用体积

### 2. **子应用独立部署与热更新**

- 支持**热更新**: 可以单独更新某个课件版本,无需更新整个播放器
- 日志中记录了 `previewVersion`,说明不同课件可能使用不同版本

### 3. **页面切换时的状态隔离与预加载**


- 游戏页和课件页**同时存在**,通过 CSS 切换显示,避免重新加载
- 游戏引擎(Cocos)初始化成本高,保持实例存活,切换时只需隐藏/显示
- 各子应用状态独立,切换页面时不会互相干扰

### 4. **跨应用通信需求**


**为什么需要微前端:**
- 主应用(mcc-player)需要向子应用(preview)传递全局数据:
  - 角色信息 (sender/receiver)
  - 初始化参数 (initParam)
  - 课件资源 (resourceData)
  - 播放速率 (playbackRate)
- 子应用需要向主应用上报状态和事件
- microApp 提供了标准化的**数据下发**和**事件监听**机制

### 5. **资源预加载与性能优化**

- 可以按需加载不同页面类型的资源,避免一次性加载所有内容

- 可以针对不同类型的子应用设置不同的错误处理策略
- 游戏应用设置 `disableSandbox: true`,说明某些场景需要绕过隔离

### 7. **多技术栈共存**

从项目结构可以看出:
- **mcc-player**: 主播放器框架 (TypeScript + Vite)
- **preview**: 课件预览端 (React 18 + Hox)
- **cocos-game**: 游戏 (Cocos 引擎)
- **bind-manager**: 绑定管理器 (独立模块)

**为什么需要微前端:**
- 不同团队可能使用不同的技术栈开发
- 游戏引擎(Cocos)与 Web 框架(React)需要独立运行环境
- 避免技术栈绑定,保持各模块的技术选型灵活性

## 总结

微前端在这个项目中的核心价值:

| 需求 | 微前端解决方案 |
|------|--------------|
| **多类型内容** | 动态加载不同子应用(course/cocos) |
| **独立部署** | 子应用通过URL远程加载,支持版本化管理 |
| **状态隔离** | 各子应用独立运行,避免互相干扰 |
| **跨应用通信** | microApp 提供标准化的数据传递机制 |
| **性能优化** | 保活模式、预加载、按需加载 |
| **错误隔离** | 子应用崩溃不影响主应用 |
| **技术栈灵活** | React、Cocos等不同技术栈共存 |

这种架构特别适合**在线教育系统**,因为课件内容多样化、更新频繁、且需要支持实时互动(信令同步),微前端提供了良好的**解耦**和**扩展性**。
