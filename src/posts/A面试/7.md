亮点在于数据结构的复杂性，不是简单的一维数组排序。
# 拖拽
---

结合代码，一步步说清楚：

---

## 第一步：展平数据（flattenArray）

后端数据是嵌套的，渲染前先展平成一维数组：

```
后端数据：
{ id: '页1', section: [{ name: '第一章' }] }
{ id: '页2' }

展平后：
[
  { type: 'section', name: '第一章', parent: { id: '页1' } },
  { id: '页1' },
  { id: '页2' },
]
```

这样 `react-beautiful-dnd` 才能正常渲染和拖拽。

---

## 第二步：拖拽开始（onDragStart）

检查被拖的是不是**展开状态的节标题**，如果是就先折叠它，让它把下面的页面"收进去"一起移动：

```
拖动"第一章"节标题（展开状态）
  → onDragStart 检测到 type=section 且 secPack=false
  → 自动调用折叠，把节标题下的页面标记 isPack=true（隐藏）
  → 节标题的 count 记录下面有几个页面
```

---

## 第三步：排序算法（reorder）

拖拽结束后计算新的列表顺序，核心逻辑：

```
移动页面时：
  sourceIndex=1 → destinationIndex=3
  向下拖 → splice(1, 1) 删掉，再 splice(3, 0, item) 插入

移动节标题时（count=2，下面有2个页面）：
  moveLen = 1(节标题) + 2(页面) = 3
  splice(sourceIndex, 3) 整段删掉
  再根据方向插入到正确位置
```

**向上 vs 向下拖的索引处理不同**：

```
向下拖：destinationIndex 要减去 moveLen 的偏移
        result.splice(destinationIndex + 1 - moveLen, 0, ...removed)

向上拖：直接插入
        result.splice(destinationIndex, 0, ...removed)
```

原因是删除后数组长度变了，向下拖时目标索引会漂移，需要补偿。

---

## 第四步：还原嵌套结构（reconstructArray）

拖拽完的扁平数组还要还原成嵌套结构，才能存给后端：

```
扁平：
[
  { type: 'section', name: '第一章' },
  { id: '页1' },
  { id: '页2' },
]

还原：
[
  { id: '页1', section: [{ name: '第一章' }] },
  { id: '页2' },
]

逻辑：遍历时遇到 section 类型，
      找到它后面最近的一个页面节点，
      把 section 挂到那个页面的 section 字段上
```

---

## 完整流程串起来

```
后端嵌套数据
  → flattenArray() 展平 → dataSource（一维数组）
  → react-beautiful-dnd 渲染列表
  → 用户拖拽
  → onDragStart：节标题自动折叠
  → onDragEnd：reorder() 重排索引
  → reconstructArray() 还原嵌套
  → setWorkspaceList() 更新状态
  → syncOriginSort() 持久化到后端
```

---

面试时这个流程说出来，**展平→排序→还原**这三步转换是最有说头的地方。
