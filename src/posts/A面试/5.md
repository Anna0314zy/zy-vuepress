### 2.2 核心设计模式 —— Behavior-Resource 模式

这是整个编辑器最核心的架构设计，参考了 [阿里巴巴 Designable](https://github.com/alibaba/designable) 的设计思想：

```
┌─────────────────────────────────────────────────────┐
│                  GlobalRegistry                       │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐   │
│  │ RootBehavior │ │ ShapeBehavior│ │ ImgBehavior │   │
│  │ TextBehavior │ │ VideoBehavior│ │GameBehavior │   │
│  │AudioBehavior │ │ RichTextBeh. │ │CameraBehav. │   │
│  └─────────────┘ └─────────────┘ └─────────────┘   │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                   ToolbarMenu                        │
│  ┌──────────────┐ ┌──────────────┐ ┌─────────────┐ │
│  │ShapeResource  │ │ImgResource   │ │TextResource │ │
│  │VideoResource  │ │AudioResource │ │GameResource │ │
│  │CameraResource │ │RichTextRes.  │               │ │
│  └──────────────┘ └──────────────┘ └─────────────┘ │
└─────────────────────────────────────────────────────┘
```

**每个组件的三位一体定义**：

1.  **Behavior** (`createBehavior`) —— 定义组件在编辑器中的行为：
    -   `selector`: 匹配规则，选中节点时判断是否展示此配置表单
    -   `designerProps.propsSchema`: 属性面板的 JSON Schema 表单配置
    -   `designerProps.defaultProps`: 组件默认 props
    -   `designerProps.getComponentProps()`: 向组件注入编辑器运行时 props
    -   `designerLocales`: 中文本地化文案

2.  **Resource** (`createResource`) —— 定义组件在工具栏中的入口：
    -   `title`: 工具栏展示名
    -   `icon`: 图标标识
    -   `elements`: 拖入画布时创建的节点结构

3.  **Component** —— 编辑态渲染组件：
    -   从 [TreeContent.tsx](editor/src/components/TreeContent.tsx) 的 `ContentComponent` 注册到画布
    -   接收 `treeNodeProps`（编辑器注入的 DOM 事件/属性）+ `componentProps`（组件自身属性）

## 三、核心模块实现详解

### 3.1 设计器引擎 (Engine)

```typescript
// App.tsx
const engine = createDesigner()
```

Engine 是编辑器的核心，包含：
-   **Workbench（工作台）**：管理多个 Workspace（页面）
-   **Workspace（工作区）**：每页课件对应一个 Workspace，包含 Viewport、Operation、Tree
-   **TreeNode（节点树）**：课件内容的树形结构，支持序列化/反序列化
-   **Event System（事件系统）**：`SelectNodeEvent`、`SwitchWorkspaceEvent`、`ContextMenuEvent`、`MakeGroupNodeEvent` 等

### 3.2 页面类型系统

编辑器支持三种页面类型（定义在 `@editor/react/src/widgets/AddPageWidget`）：

| PageType | 枚举值 | 工具栏菜单 | 用途 |
|----------|--------|------------|------|
| `normalPage` | 1 | 形状、富文本、资源(图片)、视频流 | 普通课件页 |
| `videoPage` | - | 资源(视频) | 视频专用页 |
| `gamePage` | 2 | 游戏 | 游戏互动页 |

不同页面类型展示不同的工具栏菜单（见 [ToolbarData.tsx](editor/src/ToolbarData.tsx)），限制可添加的组件类型。

```typescript
// 三大变换能力
moveableProps = {
  draggable: true,    // 拖拽移动
  resizable: true,    // 缩放大小
  rotatable: true,    // 旋转角度

```

**不同组件的变换限制**（[Moveable/index.tsx](editor/src/components/Moveable/index.tsx)）：

| 组件 | 拖拽 | 缩放 | 旋转 | 等比缩放 |
|------|------|------|------|----------|
| Video | ❌ | ❌ | ❌ | - |
| Game | ❌ | ❌ | ❌ | - |
| Camera | ✅ | ✅ | ❌ | ✅ |
| Group | ✅ | ❌ | ❌ | - |
| 其他 | ✅ | ✅ | ✅ | - |

**自定义 Able**（[ables.tsx](editor/src/components/Moveable/ables.tsx)）：
-   `editable` Able：在选中元素旁渲染**删除**和**复制**按钮
-   删除时同步清理关联动画和视频任务
-   复制时偏移 10px 避免重叠

### 3.6 数据持久化

**课件数据结构**：
```
Slide
 ├── slideStructure (JSON)  // 页面顺序/类型信息
 └── Page[]
      ├── mainContentStructure (JSON)  // TreeNode 序列化
      ├── fileMd5List: string[]         // 页面引用的资源 MD5 列表
      ├── gameId?                        // 游戏页的游戏 ID
      └── gameTemplateId?               // 游戏模板 ID
```

**保存策略**（[useAppFn.ts](editor/src/hooks/useAppFn.ts)）：
-   **自动保存**：每 60 秒自动保存当前页
-   **切换页面保存**：切换 Workspace 时保存上一页
-   **手动保存**：点击保存按钮
-   **离线保存**：`beforeunload` 时通过 `navigator.sendBeacon` 发送退出编辑请求

**编辑锁**：通过 `canEdit` API 实现多人互斥编辑，进入时上锁，离开时解锁（`sendBeacon`）。

### 3.7 预览系统

预览功能（[Preview/index.tsx](editor/src/components/Preview/index.tsx)）采用 **iframe + post-me 通信**：

```
┌──────────────────┐        post-me         ┌──────────────────┐
│   Editor 窗口     │ ◄──────────────────► │   Preview iframe  │
│                  │  ParentHandshake      │                  │
│  serialize()     │  remoteHandle.call    │  渲染课件内容      │
│  pages + files   │  ("getSlideData")     │                  │
└──────────────────┘                        └──────────────────┘
```

1.  打开预览弹窗，加载预览页面 iframe
2.  iframe `load` 后建立 `post-me` 的 `ParentHandshake` 连接
3.  Editor 将序列化的所有页面数据 + 资源列表传给 iframe
4.  iframe 使用渲染模式（非编辑模式）展示课件

### 3.8 游戏系统

游戏组件通过 **iframe 嵌入**实现隔离（[editingGame.tsx](editor/src/components/Game/editingGame.tsx)）：

-   编辑态：iframe `pointerEvents: "none"`，仅选中时可交互
-   游戏库弹窗（[GameModal](editor/src/components/GameModal/index.tsx)）：全屏 Modal 内嵌游戏管理后台 iframe
-   保存游戏时通过 `useSaveGame` hook 处理游戏数据回写

### 3.9 编组系统

编组/解组逻辑（[useMarkNodeGroup.ts](editor/src/hooks/useMarkNodeGroup.ts)）：

-   **编组条件**：选中多个非 Group 且 depth===1 的节点
-   **解组条件**：选中的节点父节点为 Group
-   编组时：计算包围盒，创建 Group 节点，将子组件 transform 转为相对于 Group 的偏移
-   解组时：将子组件 transform 恢复为绝对坐标，删除 Group 节点及关联动画

### 3.10 右键菜单

[ContextMenu.tsx](editor/src/components/ContextMenu.tsx) 基于 Ant Design `Dropdown`：
-   编组操作（多选时出现）
-   解组操作（选中 Group 子元素时出现）

## 六、关键设计亮点

1.  **Behavior-Resource 解耦**：组件定义、编辑行为、工具栏入口三者分离，扩展新组件只需新增三个配置
2.  **JSON Schema 驱动表单**：属性面板完全由 Schema 描述，`x-component` 映射自定义 Setter 组件
3.  **Engine 事件驱动**：通过 `engine.subscribeTo(Event)` 监听节点选中/切换页面/右键菜单等事件
4.  **iframe 隔离**：游戏和预览均使用 iframe + post-me 通信，保证编辑器稳定性
5.  **视口缩放**：通过 `viewportPercentage` 实现画布缩放，编组/解组时坐标换算
6.  **编辑锁**：后端互斥锁 + `sendBeacon` 确保关闭页面时释放锁

## 增加技术深度的简历写法

### 一、课件页拖拽排序

> 基于 **react-beautiful-dnd** 实现课件页列表拖拽排序，针对"页面 + 节（Section）"混合数据结构设计了 **`reorder` 算法**：通过 `getPageIndex` / `getLastTagIndex` 计算页面与节的边界索引，区分向上/向下拖拽两种方向，正确处理节折叠（`secPack`）状态下的批量移动——折叠节拖拽时自动展开并携带其下所有子页面整体移动；拖拽结束后通过 `reconstructArray` 将扁平化的列表还原为嵌套结构，同步更新 `workspaceList` 状态并持久化 `slideStructure`

**关键技术点**：

| 问题 | 解决方案 |
|------|---------|
| 节与页面混合排序 | `flattenArray` 展平 → `reorder` 计算 → `reconstructArray` 还原嵌套 |
| 折叠节整体拖拽 | `onDragStart` 检测折叠态，自动展开并计算 `moveLen`，splice 时整段移动 |
| 移动到折叠节内 | 检测 `secPack` 状态，插入到节末尾并标记 `isPack: true` |
| Shift 多选批量删除 | 记录 `clickIndex`，Shift+Click 计算区间索引，收集非 section 项提交删除 |

### 二、复制粘贴系统（Ctrl+C / Ctrl+V）

> 在 **Designable 引擎的 Shortcut + KeyboardDriver 机制**之上，实现 `Ctrl+C` 将选中节点写入内存 `Clipboard`，`Ctrl+V` 调用 `TreeNode.clone()` 深拷贝节点树并插入到 `insertPoint` 之后；`clone()` 内部通过 `toJS(this.props)` 解除 MobX 响应式引用，递归克隆子节点并重新分配 `uid`，触发 `CloneNodeEvent` 保证引擎状态同步；`KeyboardDriver` 通过过滤 `isContentEditable` / `INPUT` / `TEXTAREA` 等输入场景，避免在富文本编辑时误触发快捷键

**关键链路**：

```
用户按 Ctrl+C
  → KeyboardDriver 捕获 keydown → dispatch KeyDownEvent
  → Shortcut.matchCodes([Meta/Ctrl, C]) 命中 CopyNodes
  → Clipboard.nodes = selectedNodes（引用写入）

用户按 Ctrl+V
  → 命中 PasteNodes → TreeNode.clone(Clipboard.nodes)
  → 每个节点 clone()：new uid + toJS(props) + 递归克隆子树
  → insertAfter(cloned) 插入 DOM 树 → 触发 CloneNodeEvent → 视图响应式更新
```

## react-moveable 是如何工作的

**核心原理很简单**：你给它一个 `target`（目标 DOM 元素），它就会在那个元素周围渲染出一圈控制点（可以拖拽、缩放、旋转），并通过回调函数把变换结果告诉你。

```
用户拖拽控制点
  → onDrag / onResize / onRotate 回调触发
  → 你拿到新的 transform / width / height 数值
  → 手动更新 target.style（更新 DOM 视觉）
  → 同时更新 TreeNode.props.style（更新数据）
  → MobX 响应式驱动组件重渲染
```

**它本身不管数据，只负责给你"变换后的数值"**，你自己决定怎么用这个数值。

## 这个项目里具体做了什么

| 部分 | 做的事 |
|------|--------|
| **target 怎么来的** | 监听引擎的 `SelectNodeEvent`，通过被选中节点的 id 在 DOM 里查找对应元素，传给 Moveable |
| **选框多选** | 用 `react-selecto` 实现框选，框选结束把选中的 DOM 列表传给 Moveable 的 `target` |
| **变换回调** | `onDrag/onResize/onRotate` 里同时更新 DOM style 和 `TreeNode.props.style`（双写保证视觉和数据同步） |
| **画布缩放适配** | 监听 `viewportPercentage` 变化时强制调用 `moveableRef.current.updateRect()` 重新计算控制点位置 |
| **编组处理** | Group 内的子元素拖拽时，转而更新父 Group 节点的坐标，而不是子节点 |

## 这套 SnapLine 系统是怎么工作的

整个流程分三步：

**1. 拖拽开始（dragStart）**
- 记录所有画布内可见节点的位置和尺寸，存入 `viewportRectsStore`

**2. 拖拽移动（dragMove）**
- `calcAroundSnapLines`：遍历画布所有节点，计算它们的边缘线（上下左右中心），与当前拖拽元素的边缘线做距离计算，距离 < 6px 就生成一条 SnapLine
- `calcAroundSpaceBlocks`：计算拖拽元素与周围元素的间距，用于"等间距对齐"提示
- `calcRulerSnapLines`：处理标尺手动添加的辅助线

**3. 吸附修正（translate / resize）**
- 找到所有满足阈值的 `closestSnapLines`，调用 `line.translate(node, translate)` 自动修正坐标，让元素"吸"过去

## 和 react-moveable 的关系

| | react-moveable 的 snappable | 这套 SnapLine |
|--|--|--|
| 作用范围 | 当前项目用的 MoveableContainer | 引擎自带（Designable 体系） |
| 实现方式 | 库内置，配置即用 | 手写几何计算 |
| 功能 | 元素边缘/中心对齐 + 辅助线显示 | 同上，另加等间距测量块 |

**两套系统同时存在，`react-moveable` 这套是实际在用的，`TransformHelper` 这套是引擎遗留的老代码（当前画布用的是 moveable 方案）。**

面试如果被问到，可以说："项目里有两套对齐方案，最终采用的是 react-moveable 的 snappable，引擎层也保留了一套自研的基于几何距离计算的 SnapLine 系统。"
