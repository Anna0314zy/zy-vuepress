CourseGameClient（Game-SDK）

1. 设计背景与目标

CourseGameClient 是 Course 互动课件体系的前端游戏运行 SDK。它的目标是：

  1. 统一游戏的状态管理机制（state + props）。
  2. 规范宿主端与游戏的双向通信协议（EventEmitter 消息事件）。
  3. 提供可控渲染周期（render → useEffect）。
  4. 提供游戏常用能力的封装（静音、测评、外部控制、编辑模式等）。
  5. 让游戏开发者“只写逻辑，不关心环境差异”，使游戏可以无缝运行于：老师端、学生端、预览器、编辑器等不同客户端。

从本质上讲，它是一个轻量版 React 运行时 + 消息桥接层 + 游戏生命周期调度器。

2. SDK 扮演的角色（职能）

✔ 2.1 游戏的运行容器 Runtime

它管理：

  • 游戏的“内部状态” state
  • 宿主传入的“外部属性” props
  • 状态变化触发渲染 render()
  • render 内部执行“副作用” useEffect()

类似 React，但更轻、更可控，适合嵌入式小游戏。

✔ 2.2 Host（宿主端）与游戏之间的通信桥

通过继承 EventEmitter，sdk 提供规范事件：

宿主 → 游戏：

事件名含义
pullState主动下发 state 给游戏
pullProps下发 props
mute控制游戏静音
beginSpeechEvaluation开始语音测评
onSpeechEvaluationResult回传测评结果
onSpeechEvaluationError测评失败

游戏 → 宿主：

事件名含义
pushState游戏告诉宿主“我的 state 更新了”
pushProps游戏请求宿主更新 props
beginSpeechEvaluation_action游戏请求宿主开始测评
stopSpeakingTest_action游戏请求宿主停止测评

结论：SDK 是游戏与宿主之间的“协议层”。

✔ 2.3 内置游戏常用能力

SDK 内置：

  • 🔊 静音 onMute(callback)
  • 🎤 口语测评 beginSpeechEvaluation / stopSpeakingTest
  • 🛠 编辑模式 isEdit
  • 🔄 同步渲染 sync()

这些能力屏蔽了跨端差异，使游戏在任何运行环境都能表现一致。

✔ 2.4 游戏生命周期管理（类似 React Runtime）

SDK 明确规定渲染流程：

setState()
→ 触发 onStateChange()
→ render()
→ 执行 useEffect()

并阻止在 useEffect 中再次 setState（避免无限循环）：

if (this.isDispatching) {
  throw new Error('do not setState in useEffect')
}

这就是“受控渲染模型”。

3. 核心设计理念

3.1 “状态驱动渲染”

所有 UI 更新都来自两种输入：

  • state：游戏自己的内部状态
  • props：宿主端传来的外部参数

SDK 保证：

  • 不直接操作 DOM（避免 iframe 差异问题）
  • 让开发者只关心 state → render()

简化了游戏逻辑。

3.2 “Host / Client 解耦通信”

宿主端不知道游戏内部结构，但通过事件即可控制游戏。

游戏也不知道宿主端运行在哪（PC、iPad、小程序），但通过事件就能与宿主交互。

协议化 → 解耦 → 低成本维护

3.3 “可观察、副作用隔离”

SDK 内部模拟一个简化版 hooks：

useEffect(callback, deps)

作用：

  • 数据变化才执行副作用
  • 自动执行清理函数
  • 存储依赖数组 dep → diff

比 React 更小，但对小游戏逻辑已经足够。

4. useEffect 实现机制解析

整体流程

  1. 记录 deps：effectDeps
  2. 检查是否变化
  3. 如变化，则执行 cleanup()
  4. 执行 callback()
  5. 保存 cleanup 回调
  6. 每次渲染前 effectIndex 重置为 0

意味着 render 内部调用 useEffect 的顺序必须稳定（和 React 一样）。

5. 生命周期流程图

宿主传入 props/state
        ↓
事件触发 pullProps / pullState
        ↓
props/state setter
        ↓
onStateChange()
        ↓
render()
        ↓
useEffect()
        ↓
（游戏内部画面更新）

游戏内部每一次更新都遵循这条链路，所有行为都有源头，可记录、可调试、可回放。

6. API 说明

6.1 状态相关

API描述
setState(partial)合并内部 state，并通知宿主
state getter获取内部 state（不存在则创建默认值）
state setter设置完整 state，并触发渲染

6.2 属性 props

props 由宿主驱动：

client.on('pullProps', props => { this.props = props })

游戏不可直接修改 props，只能请求：

setProps({ ... })

宿主再决定是否接受（单向数据流）。

6.3 副作用 useEffect

与 React 类似，用于：

  • 监听 props/state 变化
  • 注册事件
  • 执行动画
  • 绑定外部资源

6.4 宿主能力封装

静音

onMute(cb)

语音测评

beginSpeechEvaluation(content, options?)
stopSpeakingTest()

并接受宿主返回：

onbeginSpeechEvaluation
onStopSpeakingTest
ononSpeechEvaluationError

编辑模式

this.isEdit = true/false

7. init() 注册机制

static init() {
  window.CourseGameSdk = {
    getGameClient: () => window.__CREATE_GAME_INSTANCE__
  }
}

使外部任何端都可以：

window.CourseGameSdk.getGameClient()

从 iframe 内拿到当前游戏实例，实现跨 iframe 的“唯一实例访问”。

8. 为什么这样设计？

👍 8.1 比 React/Vue 更轻

小游戏无需完整框架，仅需：

  • 状态
  • 渲染周期
  • 副作用
  • 通信

SDK 就是只保留必要部分的“React Runtime”。

👍 8.2 强制所有变化都有来源（可追踪）

链路：

setState → onStateChange → render → useEffect

任何 UI 问题都能定位到 state 流程，无“野生逻辑”。

👍 8.3 跨端一致性

各平台表现一致，因为：

  • 没有直接 DOM 操作
  • 用事件协议桥接宿主
  • 生命周期可控

👍 8.4 方便编辑器接入

编辑器可以随时更改 props/state，触发游戏即时刷新。

👍 8.5 游戏逻辑不与平台绑定

游戏代码不需要知道运行在：

  • Web 端
  • 编辑器
  • App 内嵌引擎
  • 小程序

只需和 SDK 交互。

9. 典型使用方式（示例）

游戏逻辑：

const client = new CourseGameClient()

client.getDefaultState = () => ({
  score: 0,
})

client.render = () => {
  const { score } = client.state

  client.useEffect(() => {
    console.log('score changed!', score)
  }, [score])
}

宿主发送数据：

game.emit('pullState', { score: 10 })

游戏请求宿主：

client.setState({ score: client.state.score + 1 })

总结（一句话）

CourseGameClient 是一个：

👉 轻量级 React-like 渲染引擎 + 游戏事件协议层 + 宿主能力封装层

它让游戏具有统一、可控、跨端一致的运行时。
